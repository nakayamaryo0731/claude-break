## 2. なぜ cc-fit なのか

### 課題

- Claude Code にタスクを依頼すると、数十秒〜数十分の待ち時間が発生する
- この間、エンジニアは「終わったかな？」とターミナルをチラチラ見るだけで、時間を無駄にしがち
- **長時間座りっぱなし、水分不足、運動不足はエンジニアの職業病**。WHO は「座りすぎ」を喫煙に次ぐ健康リスクと位置づけている
- AI コーディングツールの普及で「人間が手を動かさない時間」はさらに増加傾向

### 既存の類似ツール

| ツール | 特徴 | cc-fit との違い |
|---|---|---|
| **cc-spinner** | Claude Code のスピナー表示をアニメのセリフにカスタマイズ | 見た目の変更のみ。行動提案なし |
| **Super Productivity** | タスク管理 + 休憩リマインダー | Claude Code 連携なし。汎用ツール |
| **Stretchly** | 定期的にストレッチを通知 | 固定間隔。AI作業の開始/終了に連動しない |

### cc-fit の差別化

**「AIが働いてる間に人間が健康になる」という逆転の構図**がキャッチー。単なる休憩リマインダーではなく:
- **健康特化**: 家事やタスク管理ではなく、ストレッチ・運動・水分補給・目の休憩に絞る
- **Claude Code 連動**: AIの作業開始/完了に自然に連動。固定間隔の通知より文脈に合っている
- **エンジニアの職業病へのアンサー**: 座りすぎ・目の疲れ・水分不足という、ターゲットが全員抱える問題に直接アプローチ

## 4. コアフロー

```
ユーザーが Claude Code にプロンプト送信
        |
    [UserPromptSubmit hook 発火（async: true）]
        |
    cc-fit start: 既存タイマーがあればkill → 新タイマー開始
    （開始時刻は state.json から引き継ぎ。初回なら現在時刻）
        |
    30秒経過 → 通知「肩回し10回やろう」 + log.jsonl に記録
        |
    さらに1分経過 → 「スクワット10回いける？」
        |
    さらに1分経過 → 「コップ1杯水飲もう」
        |
    [Stop or Notification hook 発火（async: true）]
        |
    cc-fit stop: 5秒デバウンス開始
        |
    5秒以内に start が来た → デバウンスキャンセル、タイマー継続
    5秒経過 → タイマー停止 + 通知「Claude 待ってるよ！戻ってきて」
```

### 設計判断: なぜ「時間見積もり」をしないのか

当初はタスク実行時間を事前に見積もる方式を検討した。しかし以下の理由で断念:

1. **予測精度の本質的な限界**: 同じ「バグ修正して」でも1行で終わる場合と30分かかる場合がある
2. **途中の割り込み**: ユーザー認証要求、選択肢形式の質問など、予測不可能な中断が発生する
3. **外れた予測は逆効果**: 「5分」と言われて洗い物を始めたら10秒後に終わった、は最悪の体験

代わりに**ストップウォッチ型**を採用:
- 事前予測なし。「Claudeが動いている」状態を検知するだけ
- 経過時間に応じてアクティビティを提案（すべて2分以内で完了するもの）
- いつClaudeが終わっても問題ないアクティビティのみ提案する

## 5. MVP 要件定義

### 5.1 機能要件

#### コア機能（MVP必須）

| # | 機能 | 詳細 | 優先度 |
|---|---|---|---|
| F1 | **タスク開始検知** | Claude Code の `UserPromptSubmit` hook でタスク開始を検知 | P0 |
| F2 | **タスク完了検知** | `Stop` hook でタスク完了を検知。`Notification` hook で入力待ち（権限確認、質問等）を検知 | P0 |
| F3 | **アクティビティ提案** | タスク実行中、経過時間に応じてmacOS通知でアクティビティを提案 | P0 |
| F4 | **呼び戻し通知** | タスク完了 or 入力待ちになったら「戻ってきて」通知 | P0 |
| F5 | **カテゴリ設定** | ユーザーがアクティビティカテゴリを設定ファイルで定義可能 | P0 |
| F6 | **セットアップコマンド** | `cc-fit init` で Claude Code の hooks 設定を自動注入 | P0 |

#### 拡張機能（MVP後）

| # | 機能 | 詳細 | 優先度 |
|---|---|---|---|
| F7 | **健康ダッシュボード** | 「今日のストレッチ回数」「水を飲んだ回数」「スクワット合計回数」等の健康統計表示 | P1 |
| F8 | **カスタム通知間隔** | 提案の間隔をユーザーが調整可能 | P1 |
| F9 | **サウンド通知** | 呼び戻し時に音を鳴らすオプション | P1 |
| F10 | **OS固有の通知カスタマイズ** | OS毎の通知スタイル調整（アイコン、サウンド等） | P2 |

### 5.2 アクティビティのデフォルトプリセット（健康特化）

| カテゴリ | アクティビティ例 | 所要時間 | 健康効果 |
|---|---|---|---|
| **ストレッチ** | 肩回し10回、首のストレッチ、背伸び、前屈、手首回し | 15-30秒 | 肩こり・腰痛予防 |
| **筋トレ** | スクワット10回、腕立て5回、プランク30秒、カーフレイズ20回 | 30-60秒 | 筋力維持・血行促進 |
| **水分補給** | コップ1杯水を飲む、白湯を飲む | 15-30秒 | 脱水予防・集中力維持 |
| **目の休憩** | 窓の外を20秒見る（20-20-20ルール）、目を閉じて10秒、遠くを見る | 20-30秒 | 眼精疲労・ドライアイ予防 |
| **呼吸** | 深呼吸3回、4-7-8呼吸法、ボックス呼吸 | 30-60秒 | 自律神経調整・ストレス軽減 |
| **姿勢リセット** | 座り直し、骨盤を立てる、胸を開く | 10-15秒 | 猫背・腰痛予防 |

ユーザーは設定ファイルでカテゴリを自由に追加・削除・編集できる。

### 5.3 通知タイミングロジック

```
タスク開始
  |
  +-- 0〜30秒: 提案なし（短いタスクにはいちいち通知しない）
  |
  +-- 30秒経過: 1回目の提案（目の休憩、深呼吸、姿勢リセット系）
  |
  +-- 1分30秒経過: 2回目の提案（ストレッチ、水分補給系）
  |
  +-- 3分経過: 3回目の提案（筋トレ系）
  |
  +-- 以降2分間隔で提案（同じカテゴリが連続しないようシャッフル）
  |
  +-- タスク完了 or 入力待ち: 「戻ってきて！」通知
```

### 5.4 非機能要件

| # | 要件 | 詳細 |
|---|---|---|
| NF1 | **軽量** | 常駐プロセスなし。Hook 経由のイベント駆動で動作 |
| NF2 | **セットアップの簡単さ** | `npx cc-fit init` 一発で設定完了 |
| NF3 | **Claude Code への影響ゼロ** | Hook のオーバーヘッドを最小限に。タスク実行を遅延させない |
| NF4 | **クロスプラットフォーム対応** | node-notifier により macOS / Linux / Windows をMVPから対応。OS分岐はライブラリに委譲 |

## 6. 技術アーキテクチャ

### 6.1 全体構成

```
Claude Code
    |
    | hooks（settings.json に設定）
    |
    v
cc-fit CLI（Node.js）
    |
    +-- cc-fit start（async hookとして実行、プロセス分離不要）
    |      |
    |      +-- PIDファイル（~/.cc-fit/timer.pid）で排他制御
    |      +-- 開始時刻を state.json に記録・引き継ぎ
    |      +-- 経過時間に応じて node-notifier で通知を発行
    |      +-- アクティビティをカテゴリからランダム選択
    |      +-- 通知履歴を log.jsonl に記録
    |
    +-- cc-fit stop（5秒デバウンス後にタイマー停止 + 呼び戻し通知）
    |
    +-- 設定ファイル（~/.cc-fit/config.json）
           |
           +-- カテゴリ定義
           +-- 通知間隔設定
           +-- デバウンス秒数（デフォルト5秒）
           +-- say有効/無効（macOS限定、デフォルトOFF）
           +-- 有効/無効トグル
```

### 6.2 Claude Code Hooks 設定

`cc-fit init` が以下を `~/.claude/settings.json` に自動追加:

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cc-fit start",
            "async": true
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cc-fit stop --reason=completed",
            "async": true
          }
        ]
      }
    ],
    "Notification": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cc-fit stop --reason=needs-input",
            "async": true
          }
        ]
      }
    ]
  }
}
```

### 6.3 技術スタック

| レイヤー | 技術 | 理由 |
|---|---|---|
| ランタイム | **Node.js** | `npx` で即実行可能。Claude Code ユーザー（エンジニア）は Node.js 環境がほぼ確実にある |
| 通知 | **node-notifier** | OS判定を内部で行いクロスプラットフォーム対応（macOS: osascript, Linux: notify-send, Windows: toast通知）。自前でOS分岐を書く必要なし |
| タイマー管理 | **async hook + PIDファイル** | `async: true` で hook 自体がバックグラウンド実行。PIDファイルで排他制御 |
| ログ記録 | **JSONL ファイル**（`~/.cc-fit/log.jsonl`） | 通知履歴を1行1イベントで記録。将来のダッシュボード機能の土台 |
| 音声読み上げ | **`say` コマンド**（macOS限定） | デフォルトOFF。設定で有効化。デモ動画のインパクト用 |
| 設定管理 | **JSON ファイル**（`~/.cc-fit/config.json`） | シンプル。手動編集も容易 |
| パッケージ配布 | **npm** | `npx cc-fit init` でセットアップ完了 |

### 6.4 コマンド体系

```bash
# セットアップ（Claude Code の hooks を自動設定）
npx cc-fit init

# 手動でタイマー開始（hook から自動呼び出し）
cc-fit start

# 手動でタイマー停止（hook から自動呼び出し）
cc-fit stop

# 設定確認
cc-fit config

# カテゴリの一覧表示
cc-fit categories

# 一時的に無効化
cc-fit disable

# 有効化
cc-fit enable

# アンインストール（hooks 設定を削除）
cc-fit uninstall
```

## 11. 設計レビュー決定事項（2026-04-07）

| # | 論点 | 決定 | 理由 |
|---|---|---|---|
| Q1 | Stopフックのチャタリング | デバウンス5秒 | Stopは応答ターン毎に発火する。5秒で許可操作のチャタリングを吸収しつつ呼び戻し遅延を最小化 |
| Q2 | hookの実行方式 | `async: true` を使用 | Claude Codeをブロックする必要がないため、自前プロセス分離が不要になる |
| Q3 | タイマー多重起動 | PIDファイルで排他制御 + 開始時刻をstate.jsonで引き継ぎ | 連続プロンプトでタイマーリセットを防ぐ |
| Q4 | sayコマンド | MVP導入、macOS限定、デフォルトOFF | デモ動画のインパクト用途。クロスプラットフォーム対応の定番パッケージがない |
| Q5 | アクティビティログ | MVPからlog.jsonlに記録 | 実装コストほぼゼロ。将来のダッシュボードの土台＋SNS投稿用の数字が出せる |
| Q6 | 設定ファイル生成 | `cc-fit init`で一括生成 | initしたら使える、が最もわかりやすい |

## 回答の根拠

- Claude Code Hooks 公式ドキュメント: `UserPromptSubmit`、`Stop`、`Notification` の3イベントで開始/完了/入力待ちを検知可能
- cc-spinner（Zenn記事）: Claude Code の Spinner Verbs カスタマイズの実例。Hooks 活用の先行事例
- Super Productivity ブログ: 「Claude Code の待ち時間に何をするか」問題の存在を確認
- macOS の osascript: ネイティブ通知をシェルコマンドから発行可能
- npm の npx: インストールなしで即実行可能な配布方法

## 多面的な反論

- **通知疲れのリスク**: エンジニアは通知が多い環境で働いている。さらに通知を追加することは逆効果になる可能性がある。「通知OFF」のデフォルト設計や、適切な間隔設定が重要
- **Claude Code の待ち時間は短縮傾向**: モデルの高速化、ストリーミング表示の改善により、待ち時間自体が減っていく可能性。ツールの存在意義が薄れるリスク
